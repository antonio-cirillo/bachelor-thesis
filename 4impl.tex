Al giorno d'oggi lavorare con dati provenienti da diverse sorgenti è diventata una necessità in qualsiasi contesto, ancora di più nell'ambito del Calcolo Scientifico. Per venire incontro a questo tipo di necessità FLY mette a disposizione degli sviluppatori delle entità che permettono di accedere e utillizzare dati provenienti da sorgenti esterne. Attualmente, FLY permette di manipolare dati presenti all'interno di file strutturati grazie all'entità \verb|dataframe|. Essa permette di effettuare operazioni di lettura, scrittura e modifica su file CSV, fogli Excel e semplici txt. Inoltre, FLY supporta l'interazione con i database MySQL tramite la presenza dell'entità \verb|sql|, che permette di stabilire una connessione ad un database MySQL sia locale che su Cloud, e dell'entità \verb|query|, che permette di creare ed eseguire query per la manipolazione dei dati. \\
L'aumento della mole di dati e la velocità con cui essi vengono prodotti ha determinato la nascita di soluzioni incentrate sul concetto di \textbf{scalabilità}. Grazie al dimensionamento orizzontale e ai diversi schemi flessibili utilizzabili per rappresentare i dati, i database NoSQL offrono una soluzione efficace e veloce per la gestione di set di dati di grandissime dimensioni. Essa risulta essere molto vantaggiosa per ambiti come il Calcolo Scientifico dove si preferiscono soluzioni che garantiscono la disponibilità dei dati rispetto alla loro coerenza. È risultato quindi necessario implementare all’interno di FLY un supporto ai database NoSQL. Uno degli obiettivi di sviluppo principale di questo lavoro di tesi è quello di permettere allo sviluppatore di interagire con un cluster NoSQL distribuito su Cloud utilizzando la stessa sintassi necessaria per interagire con un cluster NoSQL distribuito in locale, in modo da rendere la sintassi facile da imparare e da utilizzare. Per poter realizzare questo obiettivo è risultato necessario implementare all'interno di FLY un supporto ai database NoSQL compatibili con MongoDB, in quanto al momento MongoDB è l'unico database NoSQL supportato dai diversi Cloud Provider. \\
In questo capitolo verrà descritta l'implementazione all'interno di FLY del supporto ai database MongoDB distribuiti sia in locale che su Cloud.

\section{Obiettivi}
Per permettere l'interazione con i database MongoDB all'interno di FLY sono stati posti alcuni obiettivi implementativi:
\begin{itemize}
    \item connettersi e accedere ad un cluster MongoDB distribuito sia in locale che su Cloud;
    \item creare ed eseguire query per la manipolazione dei dati;
    \item fornire metodi per la gestione dei risultati;
    \item creare un meccanismo di interazione simile a quello già presente per i database MySQL.
\end{itemize}

\section{Implementazione}
All'interno di FLY sono presenti due entità che permettono all'utente di interagire con un database MySQL:
\begin{itemize}
    \item \verb|sql| - entità che permette di stabilire una connessione ad un database di tipo MySQL;
    \item \verb|query| - entità che permette di creare ed eseguire query sul database.
\end{itemize}
Grazie all'entità \verb|query| è possibile effettuare operazioni di:
\begin{itemize}
    \item \verb|select|: query di selezione che restituisce come risultato una tabella;
    \item \verb|value|: query di aggregazione che restituisce come risultato una stringa complessa che deve essere opportunamente filtrata;
    \item \verb|update|: query di inserimento, modifica o cancellazione che restituisce come risultato il numero di righe coinvolte;
\end{itemize}
Queste due entità sono state utilizzate come punto di partenza per implementare l'interazione con i database MongoDB all'interno di FLY. In modo analogo all'entità \verb|sql| è stata implementata all'interno di FLY l'entità \verb|nosql|, necessaria per stabilire una connessione ad un cluster MongoDB. Inoltre, il comportamento dell'entità \verb|query| è stato esteso anche per i database MongoDB, in modo da creare un'interazione fra le due diverse tipologie di database il quanto più possibile omogenea.

\subsection{Connessione al database}
Come già introdotto nel capitolo precedente, la possibilità di stabilire una connessione ad un database MongoDB è stata resa possibile mediante l'implementazione dell'entità \verb|nosql|. L'obiettivo principale è quello di creare un meccanismo di connessione simile a quello già presente per i database MySQL. La dichiarazione di una variabile di tipo \verb|sql| è composta dai seguenti parametri \cite{giuseppe}:
\begin{itemize}
    \item \textbf{db\textunderscore name}: stringa rappresentante il nome del database;
    \item \textbf{resource\textunderscore  group} (solo in caso il database sia su Azure): stringa rappresentante il nome del ResourceGroup in cui è contenuta l’istanza del servizio DBaaS;
    \item \textbf{instance} (solo in caso il database sia su Cloud): stringa rappresentante il nome dell’istanza del servizio DBaaS su cui è in esecuzione il database;
    \item \textbf{user}: stringa rappresentante il nome utente per l’accesso al database;
    \item \textbf{password}: stringa rappresentante la password di accesso al database.
\end{itemize}
Per raggiungere l'obiettivo posto, l'idea implementativa dell'entità \verb|nosql| parte dall'utilizzo di questa struttura. \\ 

\subsection{Integrazione di MongoDB}
Qualsiasi nuova implementazione all'interno di FLY deve comprendere l'integrazione con l'ambiente Cloud.
Risulta quindi necessario implementare all'interno di FLY un supporto ai database NoSQL distribuiti sia in locale che su ambiente Cloud. L'implementazione deve essere tale da offrire un meccanismo di interazione indipendente dalla distribuzione del database. Per far fronte a tali necessità, all'interno di FLY è stato implementato un supporto ai database MongoDB. Questa scelta è dovuta al fatto che MongoDB è l'unico database documentale supportato dai diversi Cloud Provider. Inoltre, le librerie ufficiali MongoDB permettono di interagire con un cluster distribuito su Cloud utilizzando la stessa sintassi necessaria per interagire con cluster locale. \\
I database MongoDB, a differenza dei database MySQL, vengono partizionati in un insieme di collezioni. Una collezione può essere vista come un sotto-database indipendente dalle altre collezioni. Risulta quindi necessario introdurre un parametro, all'interno della dichiarazione dell'entità \verb|nosql|, che permette di specificare a quale collezione l'utente intende collegarsi. \\
MongoDB utilizza \textbf{Log4J} \cite{log4j}, una libreria Java sviluppata dalla Apache Software Foundation che permette di mettere a punto un sistema di logging per tenere sotto controllo il comportamento di un'applicazione, sia in fase di sviluppo che in fase di test e messa in opera del prodotto finale. Per poter quindi stabilire una connessione ad un database MongoDB è necessario specificare il path del file \textit{log4j.properties}. Questo file contiene le proprietà che verranno utilizzate per il sistema di logging del cluster MongoDB. Per far fronte a questa necessità, viene introdotto il parametro opzionale \verb|properties|, vedi Listato~\ref{lst:nosql:1}. Se non specificato, come nel Listato~\ref{lst:nosql:2}, FLY genera e utilizza in un automatico il seguende file \textit{log4j.properties}.
\begin{lstlisting}[language=Java,caption={File log4j.properties generato in automatico da FLY.}, label={lst:log}]
log4j.appender.stdout.Target=System.out
log4j.appender.stdout.layout=org.apache.log4j.PatternLayout
log4j.appender.stdout=org.apache.log4j.ConsoleAppender
log4j.rootLogger=INFO, stdout
log4j.appender.stdout.layout.ConversionPattern=%d{yy/MM/dd HH\:mm\:ss} %p %c{2}\: %m%n
\end{lstlisting}
I database MongoDB utilizzano un URI per definire le connessioni tra applicazioni e istanze MongoDB. La stringa di connessione URI include i seguenti componenti:
\begin{itemize}
    \item \textbf{mongodb://} - prefisso obliggatorio per identificare che si tratta di una stringa nel formato di connessione standard;
    \item \textbf{username:password@} - se specificato, il client tenterà di autenticare l'utente nel \textbf{authSource} \cite{authSource};
    \item \textbf{host[:port]} - l'host (e il numero di porta facoltativo) in cui è in esecuzione l'istanza MongoDB;
    \item \textbf{?$<$options$>$} - opzionale. Una stringa di query che specifica le opzioni specifiche della connessione come \verb|<name>=<value>| coppie.
\end{itemize}
La stringa di connessione URI permette quindi di specificare le credenziali di accesso da utilizzare per stabilire una connessione ad un cluster MongoDB. Per questo motivo i parametri \verb|user| e \verb|password| non sono presenti all'interno dell'entità \verb|nosql| data la loro presenza all'interno della stringa di connessione. Risulta invece necessario aggiungere i parametri \verb|collection| e \verb|properties|. \\
La dichiarazione di una variabile di \verb|type = "nosql"| è quindi composta dai seguenti parametri:
\begin{itemize}
    \item \textbf{endpoint} (solo in caso non si usano i parametri \textbf{resource\textunderscore group} e \textbf{instance}): stringa rappresentante l'endpoint del database;
    \item \textbf{resource\textunderscore group} (solo in caso il database sia su Azure): stringa rappresentante il nome del ResourceGroup in cui è contenuta l’istanza del servizio DBaaS;
    \item \textbf{instance} (solo in caso il database sia su Azure): stringa rappresentante il nome dell’istanza del servizio DBaaS su cui è in esecuzione il database;
    \item \textbf{db\textunderscore name}: stringa rappresentante il nome del database;
    \item \textbf{collection}: stringa rappresentante il nome della collezione del database;
    \item \textbf{properties} (opzionale): stringa rappresentante il path assoluto del file log4j.properties.
\end{itemize}  
L'entità \verb|nosql| offre un metodo alternativo per permettere all'utente di stabilire una connessione ad un cluster MongoDB distribuito su Azure, utilizzando i parametri \verb|resource_group| e \verb|instance|, in sostituzione al parametro \verb|endpoint|, come mostrato nel Listato~\ref{lst:nosql:3}. Questi due parametri permettono a FLY di ottenere, in modo del tutto astratto all'utente, l'endpoint del database tramite il nome dell'istanza del cluster e il nome del gruppo di risorse in cui è in esecuzione. Per poter utilizzare questi due parametri occorre utilizzare la parola chiave \verb|on| seguita dalla variabile \verb|type = "azure"|.
\begin{lstlisting}[language=FLY,caption={Dichiarazione di entità nosql che utilizza un file log4j.properties specifico}, label={lst:nosql:1}]
var dbConnLocal = [type="nosql", endpoint="endpointDatabase", db_name="nomeDatabase", collection="nomeCollection", properties="pathDelFileLog4jProperties"]
\end{lstlisting}
\begin{lstlisting}[language=FLY,caption={Dichiarazione di entità nosql per la connesione ad un database tramite l'utilizzo dell'endpoint che utilizza il file log4j.properties generato in automatico da FLY.}, label={lst:nosql:2}]
var dbConnLocal = [type="nosql", endpoint="endpointDatabase", db_name="nomeDatabase", collection="nomeCollection"]
\end{lstlisting}  
\begin{lstlisting}[language=FLY,caption={Dichiarazione di entità nosql per la connessione ad un database distribuito su Azure tramite l'utilizzo dei parametri resource\textunderscore group e instance.}, label={lst:nosql:3}]
var dbConnAz = [type="nosql",  resource_group="nomeResourceGroup", instance="nomeIstanza", db_name="nomeDatabase", collection="nomeCollection"] on CloudAz
\end{lstlisting} 

\subsection{Creazione ed esecuzione di query}
FLY mette a disposizione dell'utente l'entità \verb|query| per poter effettuare operazioni di inserimento, selezione, modifica e cancellazione sui database MySQL. La dichiarazione di una variabile di tipo \verb|query| è composta, oltre dal parametro \verb|type = "query"| che ne specifica appunto il tipo, dai seguenti parametri \cite{giuseppe}:
\begin{itemize}
    \item \textbf{query\textunderscore type}: stringa rappresentante il tipo di query da eseguire (\verb|select|, \verb|value|, \verb|update|);
    \item \textbf{connection}: nome della variabile di tipo \verb|sql| rappresentante la connessione ad un database;
    \item \textbf{query}: interrogazione in un linguaggio SQL da eseguire, può essere specificata in due differenti formati:
    \begin{itemize}
        \item stringa di testo;
        \item nome di una variabile di tipo stringa.
    \end{itemize}
    \begin{lstlisting}[language=FLY,caption={Dichiarazione di entità query per l’esecuzione di un’interrogazione di selezione che utilizza una stringa di testo.}, label={lst:a:1}]
var queryStatement = [type="query", query_type="select", connection=dbConn, query="SELECT * FROM table"]

var risultato = queryStatement.execute()
\end{lstlisting}
   \begin{lstlisting}[language=FLY,caption={Dichiarazione di entità query per l’esecuzione di un’interrogazione di selezione che utilizza variabile di tipo stringa.}, label={lst:a:2}]
var stringaSelezione = "SELECT * FROM table"
var queryStatement = [type="query", query_type="select", connection=dbConn, query=stringaSelezione]
    
var risultato = queryStatement.execute()
\end{lstlisting}
\end{itemize}
Le operazioni permesse sui database MongoDB trovano un riscontro diretto con quelle permesse sui database MySQL. Su un database MongoDB è possibile effettuare operazioni di \verb|insert|, \verb|select|, \verb|update|, \verb|delete| e, in aggiunta rispetto ai database MySQL, l'operazione di \verb|replace|. L'operazione di \verb|replace|, a differenza della semplice operazione di \verb|update|, non modifica un oggetto presente all'interno del database ma lo sostituisce con un nuovo oggetto. \\
La corrispondenza delle operazioni permesse sui diversi tipi di database permette di estendere il comportamento dell'entità \verb|query| ai database MongoDB, con una piccola differenza. Le operazioni di inserimento, modifica e cancellazione sui database MySQL vengono identificate dal parametro \verb|query_type = "update"|, in quanto la sintassi SQL specifica il tipo di operazione che si intende eseguire. Questa caratteristica permette a FLY di generare correttamente il codice Java in base al tipo di operazione che si intende eseguire. Ciò non accade con la sintassi BSON (Binary JSON), ovvero la sintassi utilizzata per interrogare i database MongoDB. Risulta quindi necessario specificare quale delle tre operazioni si intende effettuare all'interno del parametro \verb|query_type|, in modo da permettere a FLY di generare il codice Java adeguato a quella situazione. Inoltre, le operazioni di \verb|update| e \verb|replace| su un database MongoDB sono composte da due query distinte:
\begin{itemize}
    \item \textbf{filter}: query che seleziona l'oggetto da modificare all'interno del database;
    \item \textbf{query}: query che modifica l'oggetto selezionato.
\end{itemize}
Per far fronte a questo aspetto, risulta necessario introdurre un nuovo parametro, ovvero \verb|filter|, necessario nella dichiarazione di query di tipo \verb|update| e \verb|replace|. \\
L'implementazione finale di queste modifiche ha fatto sì che la dichiarazione di una variabile di tipo \verb|query| sia composta dai seguenti parametri:
\begin{itemize}
    \item \textbf{query\textunderscore type}: stringa rappresentante il tipo di query da eseguire:
    \begin{itemize}
        \item per i database MySQL i valori ammissibili sono: \verb|select|, \verb|value| e \verb|update|;
        \item per i database MongoDB i valori ammissibili sono: \verb|select|, \verb|insert|, \verb|update|, \verb|replace| e \verb|delete|.
    \end{itemize}
    \item \textbf{connection}: nome della variabile di tipo \verb|sql| o \verb|nosql| rappresentante la connessione ad un database;
    \item \textbf{filter} (solo per operazioni di \verb|update| e \verb|replace| su database MongoDB): interrogazione in linguaggio BSON per selezionare l'oggetto che si intende modificare, può essere specificata in due differenti formati:
    \begin{itemize}
        \item stringa di testo;
        \item nome di una variabile di tipo stringa.  
    \end{itemize} 
    \item \textbf{query}: interrogazione (in linguaggio SQL per database MySQL o BSON per database MongoDB) da eseguire, può essere specificata in due differenti formati:
    \begin{itemize}
        \item stringa di testo;
        \item nome di una variabile di tipo stringa;
    \end{itemize}
    Inoltre, per le operazioni di inserimento su database MongoDB, è possibile assegnare a questo parametro il nome di una variabile di tipo \verb|file|, vedi Listato~\ref{lst:test:2}. Questa funzionalità aggiunge la possibilità di utilizzare altri due parametri opzionali, Listato~\ref{lst:test:3}:
    \begin{itemize}
        \item \textbf{start}: specifica la posizione all'interno del file CSV del primo elemento da inserire nel database;
        \item \textbf{end}: specifica la posizione all'interno del file CSV dell'ultimo elemento da inserire nel database.
    \end{itemize}
\end{itemize}
Nell'introduzione del capitolo abbiamo visto come FLY supporta anche la manipolazione di dati provenienti da file strutturati come fogli Excel e file CSV. In molti casi d'uso la manipolazione dei dati comporta la necessità di salvare dati provenienti da file strutturati all'interno di un database. Per questo motivo FLY permette di specificare all'interno del parametro \verb|query| il nome di una variabile di tipo \verb|file|, come mostrato nel Listato~\ref{lst:test:2}. Questa funzionalità permette, con una semplice riga di codice, di prelevare i dati presenti all'interno di un file CSV e inserirli all'interno di una collezione specifica di un database MongoDB. Inoltre, è possibile specificare la posizione di partenza e di fine degli elementi da prelevare all'interno del file, tramite l'utilizzo dei parametri opzionali \verb|start| e \verb|end|, come mostrato nel Listato~\ref{lst:test:3}. \\
\begin{lstlisting}[language=FLY,caption={Dichiarazione di entità query per l'inserimento degli elementi provenienti da un file CSV.}, label={lst:test:2}]
var inserisciDaCSV = [type="query", query_type="insert", connection=dbConn, query=fileCSV]

inserisciDaCSV.execute()
\end{lstlisting}  
\begin{lstlisting}[language=FLY,caption={Dichiarazione di entità query per l'inserimento dei primi 150 elementi provenienti da un file CSV.}, label={lst:test:3}]
var inserisciDaCSV = [type="query", query_type="insert", connection=dbConn, query=fileCSV, start="0", end="150"]

inserisciDaCSV.execute()
\end{lstlisting} 

\subsubsection{Interagire con i risultati}
L'operazione di \verb|select|, a differenza delle altre operazioni, non produce un effetto all'interno del database, ma restituisce un set di dati che l'utente può manipolare. Una query di selezione su un database MySQL, Listato~\ref{lst:sql}, ritorna come risposta una tabella contenente i dati richiesti. Questo modello di rappresentazione è facile da implementare data la corrispondenza con il modello utilizzato dai database relazionali. 
\begin{lstlisting}[language=FLY,caption={Dichiarazione di entità query per l’esecuzione di un’interrogazione di selezione che utilizza una stringa di testo.}, label={lst:sql}]
var queryStatement = [type="query", query_type="select", connection=dbConn, query=" SELECT * FROM table"]

var risultato = queryStatement.execute()
\end{lstlisting}
Applicare questa strategia anche per i database MongoDB risulta non essere efficace. I database MongoDB sono database non strutturati, questo significa che i dati memorizzati all'interno di esso non seguono, necessariamente, la stessa struttura. Il Listato~\ref{lst:tabelle:1} mostra un esempio di dati contenuti all'interno di un database MongoDB.
\begin{lstlisting}[language=json,caption={Esempio di elementi presenti all'interno di un database MongoDB.}, label={lst:tabelle:1}]
{
	"id": 1,
	"nome": "Antonio",
	"cognome": "Cirillo"
	"anni": 21
},
{
	"id": 2,
	"luogo": "Fisciano",
	"temperaturaMax": 21,
	"temperaturaMin": 12
},
{
	"id": 3,
	"luogo": "Battipaglia",
	"temperaturaMax": 32,
	"temperaturaMin": 18
}
\end{lstlisting}
È facile notare come i tre elementi non seguono la stessa struttura. In particolare però l'elemento \verb|"id": 2| e l'elemento \verb|"id": 3| sono composti dagli stessi parametri (\verb|id|, \verb|luogo|, \verb|temperaturaMax| e \verb|temperaturaMin|). Da quest'osservazione risulta quindi impossibile unire i tre elementi all'interno di una singola tabella, ma risulta invece possibile unire l'elemento \verb|"id": 2| e l'elemento \verb|"id": 3| all'interno della stessa tabella, e isolare l'elemento \verb|"id": 1| all'interno di un'altra tabella, come riportato di seguito. 
\begin{center}
\begin{tabular}{||c c c c||} 
 \hline
 id & luogo & temperaturaMax & temperaturaMin \\ [0.5ex] 
 \hline\hline
 2 & Fisciano & 21 & 12 \\ 
 \hline
 3 & Battipaglia & 32 & 18 \\
 \hline
\end{tabular}
\end{center}
\begin{center}
\begin{tabular}{||c c c c||} 
 \hline
 id & nome & cognome & anni \\ [0.5ex] 
 \hline\hline
 1 & Antonio & Cirillo & 21 \\ 
 \hline
\end{tabular}
\end{center}
Quest'analisi ha portato all'implementazione del metodo \verb|__generateTable-| \verb|FromNoSQLQuery| all'interno del generatore di FLY. Grazie all'utilizzo di questo metodo i dati ottenuti tramite l'operazione di selezione su un database MongoDB vengono raggrupparti all'interno di un insieme di tabelle e restituite all'utente, vedi Listato~\ref{lst:tabelle:2}.
\begin{lstlisting}[language=FLY,caption={Stampa a schermo di tutte le tabelle ottenute tramite l'esecuzione di una query di selezione su un database MongoDB.}, label={lst:tabelle:2}]
var dbConn = [type="nosql", endpoint="endpointDatabase", database="nomeDatabase", collection="nomeCollection"]

var selezione = [type="query", query_type="select", connection=dbConn, query="Sintassi BSON"]
		
var listaTabelle = selezione.execute()
	
for tabella in listaTabelle {
	println "\\n" + tabella
}
\end{lstlisting}
Tramite l'utilizzo di un \verb|for|, come mostrato al rigo 7, l'utente può iterare il risultato ottenuto dall'operazione di selezione. In questo modo lo sviluppatore può interagire con le varie tabelle e quindi con gli elementi al proprio interno. \\
Oltre all'operazione di selezione, anche l'operazione di eliminazione restituisce un risultato dopo la sua esecuzione, ovvero un numero intero pari al numero di oggetti eliminati.
\begin{lstlisting}[language=FLY,caption={Stampa a schermo degli elementi eliminati da un'operazione di eliminazione.}, label={lst:tabelle:3}]
var dbConn = [type="nosql", endpoint="endpointDatabase", database="nomeDatabase", collection="nomeCollection"]

var eliminazione = [type="query", query_type="delete", connection=dbConn, query="Sintassi BSON"]
		
var oggettiEliminati = eliminazione.execute()
	
println "Numero di oggetti eliminati: " + oggettiEliminati
\end{lstlisting}
Le operazioni di \verb|insert|, \verb|update| e \verb|replace| non restituiscono nessun risultato all'interno del programma ma producono solo un effetto all'interno del database.

\section{Generazione codice Java} \label{java}
Il programma Java generato dal compilatore FLY utilizza il driver ufficiale \textit{MongoDB Java} \cite{mongodbjava} che consente alle applicazioni Java di connettersi e lavorare con i dati in MongoDB. Per permettere al programma Java di utilizzare la libreria ufficiale di MongoDB, è stata inserita la dipendenza alla libreria all'interno del file POM di Maven. \\
Nel Listato~\ref{lst:java:1} è mostrato il codice generato alla dichiarazione di una variaibile di tipo \verb|nosql|. Esso è utilizzabile indipendentemente dall'ambiente di esecuzione del database in quanto è sufficiente conoscere l'endpoint per accedervi.
\begin{lstlisting}[language=Java,caption={Codice Java generato per la variaibile di tipo nosql.}, label={lst:java:1}]
[...]
PropertyConfigurator.configure("log4j.properties");

MongoClient dbConnClient = new MongoClient(new MongoClientURI("endpointDatabase"));
MongoCollection <Document> dbConn = dbConnClient.getDatabase("nomeDatabase").getCollection("nomeCollection");
\end{lstlisting}
Nei capitoli successivi vedremo come l'interazione con un cluster MongoDB all'interno dei linguaggi JavaScript e Python risulta essere omogenea all'interazione tramite Java. Questo aspetto è dovuto dall'utilizzo della libreria ufficiale di MongoDB all'interno di tutti e tre i linguaggi. \\
In Java, JavaScript e Python le interrogazioni sul database sono state implementate tramite l'utilizzo di alcuni metodi presenti all'interno della libreria ufficiale di MongoDB, ovvero: 
\begin{itemize}
    \item \verb|find(query)|, per selezionare un set di oggetti all'interno di una collezione;
    \item \verb|insertMany(documents)|, per inserire più elementi in una collezione (nello sviluppo di questa tesi, questo metodo viene adattato in modo da permettere l'inserimento anche di singoli oggetti);
    \item \verb|updateMany(filter, query)|, per aggiornare tutti gli elementi di una collezione che corrispondono al filtro specificato;
    \item \verb|replaceOne(filter, query)|, per sostituire un singolo oggetto all'interno di una collezione;
    \item \verb|deleteMany(query)|, per rimuovere tutti gli oggetti all'interno di una collezione che corrispondono al filtro utilizzato.
\end{itemize}
